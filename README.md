# 📚 《遥远的救世主》读书小精灵 (RAG Knowledge Assistant)

一个基于 **Local LLM** 和 **Advanced RAG** 技术的智能读书助手，专为深度研读豆豆著作《遥远的救世主》而设计。

本项目不仅仅是一个问答机器人，更是一个**可观测、可调节、具备自我反思能力**的检索实验平台。它融合了丁元英式的“直击本质”思维与严谨的学术引用规范，通过 **Parent-Child 索引**、**动态上下文窗口** 和 **Self-RAG** 等前沿技术，帮助用户在海量文本中快速定位核心逻辑与文化属性。

> 让 AI 带你参悟强势文化的逻辑闭环。

## ✨ 核心特性

### 🧠 进阶检索引擎 (Advanced RAG Engine)
- **Parent-Child 索引策略**：
  - **子块检索**：将文本切分为 400 字的小片段进行向量嵌入，确保检索精度。
  - **父块生成**：检索命中后，自动回溯并加载对应的 2000 字完整父段落作为上下文，解决碎片化信息缺乏语境的问题。
  - **本地映射存储**：使用轻量级 `LocalFileStore` 维护父子文档映射关系，支持断点续传构建。
- **并行混合检索 (Parallel Hybrid Retrieval)**：
  - 同时执行 **BM25** (关键词匹配) 与 **Vector Search** (语义向量) 检索。
  - 结果合并后自动去重，确保既不漏掉专有名词（如“丁元英”、“古城”），也能理解深层语义（如“文化属性的本质”）。
- **智能重排序 (Smart Rerank)**：
  - 引入 Cross-Encoder 模型 (`BAAI/bge-reranker-large`) 对初筛结果进行精细打分。
  - 支持动态阈值过滤，剔除低分噪声，只保留高相关性片段。
- **动态上下文窗口 (Dynamic Context Window)**：
  - 系统根据**问题复杂度**（长度、关键词）和**检索分数分布**，智能微调送入 LLM 的文档数量（基准值 ±2）。
  - 复杂问题自动扩充上下文，简单问题或低分情况自动收缩，平衡效果与显存占用。
- **Self-RAG 闭环评估**：
  - 内置质量评估模块。检索后先通过分数或小模型 (`qwen3:0.6b`) 判断相关性。
  - 若质量不达标，自动触发**重试机制**或**降级策略**（禁用多路查询，直接使用原问题），确保回答有据可依。

### 🎨 交互式 Web 界面 (Streamlit UI)
- **全链路可视化监控**：独创的 **Pipeline 节点监控条**，实时展示 `语义重写` -> `多路发散` -> `混合检索` -> `重排序` -> `质量评估` -> `文章生成` 每个阶段的耗时。
- **动态瓶颈高亮**：自动识别耗时最长的环节（瓶颈节点），并以红色高亮显示，提供直观的调优建议。
- **实时参数调优**：用户可在界面上直接调整 `TopK`、`重排阈值`、`改写策略`、`动态窗口开关` 等核心参数，即时观察对检索效果的影响，无需重启服务。
- **证据溯源面板**：右侧边栏实时展示被引用的原文片段，区分“核心依据”与“辅助参考”，支持章节定位，让每一个结论都经得起推敲。
- **流式去重与禅意体验**：
  - 内置智能去重算法，自动过滤 LLM 重复用户问题的首行废话。
  - 加载与思考过程中融入《遥远的救世主》经典语录与禅意思考动画，沉浸感拉满。

## 🛠️ 技术栈

| 组件 | 技术选型 | 说明 |
| :--- | :--- | :--- |
| **大语言模型** | `Qwen3` (via Ollama) | 主模型负责推理与生成；`qwen3:0.6b` 负责快速评估与摘要 |
| **嵌入模型** | `BAAI/bge-large-zh` | 高精度中文向量模型，构建语义空间 |
| **重排序模型** | `BAAI/bge-reranker-large` | Cross-Encoder 架构，精准评估 Query-Doc 相关性 |
| **向量数据库** | `ChromaDB` | 轻量级、持久化向量存储，支持增量索引 |
| **存储映射** | `LocalFileStore` (JSON) | 自定义本地文件系统，存储 Parent-Child 映射关系 |
| **应用框架** | `Streamlit` + `LangChain` | 快速构建数据驱动的 Web 交互界面 |
| **检索算法** | `BM25` + `Cosine Similarity` | 并行混合检索，兼顾精确匹配与模糊语义 |
| **知识库源** | 《遥远的救世主》.txt | 豆豆著，关于文化属性与强势文化的经典著作 |

## 🚀 快速开始

### 1. 环境准备

确保您的系统已安装 **Python 3.11+** 和 **Ollama**。

#### 安装 Ollama 并拉取模型
```bash
# 访问 https://ollama.com 安装 Ollama
# 拉取所需模型 (注意：需确保本地有足够显存/内存)
ollama pull qwen3
ollama pull qwen3:0.6b  # 用于快速评估的小模型
```

### 2. 创建python虚拟环境

```bash
# 创建python虚拟环境
/opt/homebrew/bin/python3.11 -m venv .venv
```

### 3. 修改启动文件

```bash
# 授权启动文件可双击执行。
chmod +x 启动专家系统.command
```
*双击运行文件： 启动专家系统.command*

*首次启动时，系统会自动构建 Parent-Child 向量索引。由于涉及 LLM 生成摘要，可能需要几分钟时间。系统支持断点续传，中途关闭后再次启动会自动从上次进度继续。*

## ⚙️ 核心参数详解

在 Web 界面的“链路监控与调优”面板中，您可以实时调整以下参数：

| 参数名 | 含义 | 推荐值 | 影响 |
| :--- | :--- | :--- | :--- |
| **改写策略 (rewrite_mode)** | Query 重写模式 | `deep` | `direct`: 原样查询; `light`: 轻量清洗; `deep`: 深度语义扩充 (调用 LLM) |
| **发散路数 (multi_query_count)** | 多路查询变体数量 | `1` | 生成多个角度的查询语句并行检索，提高召回率，但显著增加耗时 |
| **初召 TopK (top_k_initial)** | 混合召回数量 | `8` | 从向量库和 BM25 中初步召回的文档总数 (子块) |
| **重排门槛 (rerank_threshold)** | 相关性过滤阈值 | `0.5` | 低于此分数的片段将被直接丢弃，宁缺毋滥 |
| **最大重试 (max_self_rag_attempts)** | 自动重试次数 | `1` | 检索质量不佳时自动重新检索的次数。设为 0 可禁用以提升速度 |
| **评估宽容度 (eval_top_k)** | 评估采样数量 | `3` | 评估阶段采样的文档数量，数值越大越容易通过评估 (减少重试) |
| **终选 TopK (top_k_final)** | 最终送入上下文片段数 | `3` | 经过 Rerank 后，最终交给 LLM 阅读的**父文档**基准数量 |
| **动态窗口 (dynamic_window)** | 智能微调开关 | `ON` | 开启后，系统会根据问题复杂度自动调整终选 TopK (±2) |

## 📊 可观测性与日志

系统内置了完善的日志记录机制，所有数据均本地保存，隐私安全：
- **控制台日志**：实时输出各阶段耗时、检索得分、动态窗口调整原因、重试触发情况。
- **JSONL 会话日志**：保存在 `logs/rag_session_YYYYMMDD.jsonl`。
  - 记录每次问答的完整上下文、引用情况、性能指标。
  - 自动标记潜在风险：
    - `hallucination_risk`: 生成了回答但无引用来源。
    - `low_utilization`: 检索了大量文档但引用率极低。
    - `high_latency`: 单次响应超过 15 秒。

## 💡 设计理念

1. **拒绝废话**：Prompt 工程强制模型开篇即结论，严禁“根据上下文”、“综上所述”等冗余表述；代码层自动去除重复首行。
2. **文雅排版**：关键概念加粗，原文引用使用块引用格式，并强制标注 `(见依据 N)` 和 `—— [第 X 章]`。
3. **分数优先，动态调整**：在评估环节，优先信任 Rerank 模型的数值评分；上下文长度不再是固定值，而是随问题复杂度动态呼吸。
4. **透明可控**：将黑盒的 RAG 过程白盒化。用户不仅能看到结果，还能看到哪个环节是瓶颈，并能亲手干预每一个超参数。
5. **鲁棒性设计**：支持索引构建断点续传，支持检索失败自动降级重试，确保系统在本地环境的不稳定性下依然可用。
